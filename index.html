Master Data Synchronization

Requirements

  1. Various tables of master data are needed for Production, Shipping and in some cases 3rd party Weigh Scale (WeighWiz) software
  2. Data must be kept up to date through daily extracts
  3. For some tables (Order Header, Order Items and Order Notes) more frequent extracts are needed as data changes more often

Current Solution:

  1. To meet requirement #1, various JDE Orchestrations were created to allow data to be downloaded in a specific format. See the specific list of tables below.
  2. To keep data up to date, a Full Extract is done 3x a day to the MiddleTier server and SSIS packages then extract it to the local mill. The data is Read-Only on the WMS side (maintained in JDE)
  3. For Incremental Extracts, the middletier server is crucial as it allows for a ProcessedFlag so that SSIS packages can compare data and add, update or delete as needed

Limitations:

  • In rare cases, an order can be udpated while the loading/shipping is in progress. This can result in the final shipment being incorrect/out of sync

PLC Integration

Requirements

  1. Ability to receive and store data from PLC in various formats (sometimes just a weight, sometimes a product code and other times a unit code)
  2. A function to convert product attributes to a JDE product code
  3. For some mills, a table to store WIP (Work in Progress) "Cloud" (total boards for each thickness, standard size)

Current Solution:

  1. There are 4 different tables that are currently used across various mills
    1. PLCProductionQueue - PLC looks up product data then creates an actual unit code
    2. NBInOutEntry - PLC creates a record with the product code
    3. StaplerData - PLC create a record with product code
    4. PLCWeights - PLC create a record with a weight
    Mill specific data #TODO#
     
  2. An SQL Stored Procedure exists to convert product attributes to a finished JDE product (or an error if it finds 0 or more than 1)
  3. A few tables are used to track the WIP "Cloud". The Production Line has data on how many "pcs" should be added to the Cloud (see math below). This amount is added to the PLCWIPProducedPieces for a given thickness (product ID)
    1. ProductionLine - This stores how many Press Openings for a Production Line and how many panels come from a master panel (so, for example, if a line has 14 press openings and 6 panels from a master panel, when it triggers WIP Production the total number of pieces created would be 14 * 6 = 84)
    2. PLCWIPProducedPieces - This is the "Inventory" or total number of WIP for each PLCWIPProductID
    3. PLCWIPProcessLog - An audit history of adding or removing WIP to the table above (with adjustment codes to show where it came from)

Label Printing

Requirements

  1. Ability to design and store label templates/formats (in some cases specific formats for specific products)
  2. Ability to automate label printing (not all mills) from the PLC data
  3. Ability to select product attributes to print a Production Label
  4. Ability to create production without printing a label OR to print a label without creating production records
  5. Ability to store Production data with unit codes as labels are printed
  6. Ability to reduce WIP Inventory as labels are printed

Current Solution:

  1. Currently ZPL (Zebra Printer Language) is used, and most products and mills have a standardized format
    JAS and Webstock products have their own label format, but all customized label template info is stored in the database (with the exception of 100DZWFCW which is hardcoded in the Label Desktop Config)
  2. At some mills, a desktop (Label Desktop) can be set to automated mode so it continually reads the appropriate PLC table and looks for records to consume and generate a Label
  3. When automated mode is turned off, or for mills that do not have PLC Integration, the Label Desktop presents users with a form to select product attributes and print a label
  4. When manually selecting attributes, the Label Desktop allows the user to create production only (no label) or create a label only (no production)
  5. When a label is printed, either automatically or manually (with production) a Production entry is created
    At this point, the production entry remains unverified and the Production Status is still 1
  6. WIP is reduced based on the setup linking JDE Products to WIP Products. This is currently accomplished using a few tables:
    1. Products - in JDE WIP products are created and Finished Products are then given a WIPProductCode
    2. PLCWIPProducts - on the WMS side, a unique WIPProductID links a PLCCode to the JDE WipProductCode
    3. PLCWIPBuckets - Sometimes a product may not have a WIPProductCode so the thickness is linked to a PLCCode in this table. This only works when looking up the PLCCode using the final Product
    4. PLCWIPProducedPieces - This is the "Inventory" or total number of WIP for each PLCWIPProductID
    5. PLCWIPProcessLog - An audit history of adding or removing WIP to the table above (with adjustment codes to show where it came from)

Inventory Storage

Requirements

  1. Ability to create warehouse locations
  2. Ability to scan units using a handheld device to verify production
  3. Ability to store inventory by scanning a unit and then scanning an inventory location

Current Solution:

  1. Warehouse locations are created in the database and a large barcode label is hung over the physical area
  2. At the end of the production line, after a label is printed, the production entry is unverified. Scanning it with a handheld device verifies the production (and records which user scanned it, and when)
  3. Currently, after verifying production, the unit is moved to a warehouse location and the location barcode is scanned. This creates an inventory record #TODO# (inventory table) in the database, linking that unit to a particular inventory location

Inventory Management

Requirements This is functionality for the handheld scanners

  1. Ability to create a production label from a handheld device and audit user that created it
  2. Ability to verify a production label using a handheld device (prior to putting it into inventory)
  3. Ability to create a missing unit label by either
    1. Cloning an existing label (using the unit code)
    2. From the handheld history of units scanned (also a unit code)
    3. Selecting product attributes
  4. Ability to move inventory from one location to another
  5. Ability to change a product on a label (i.e. invalidate the old label and print a new label with the new product code)
  6. Ability to invalidate a label (deletes inventory)
  7. Ability to conduct an inventory count
  8. At some mills, the ability to do a stock transfer (a type of adjustment that moves inventory out of the mill and into the virtual reload)

Current Solution:

  1. If required, production labels can be printed from the handheld device (this is different from reprinting a label since this creates a production record)
  2. After a production label is created, it must be verified. There is a specific menu option on the handheld for this.
  3. If a new label is needed, a "Print Missing Unit Label" is available which allows the user to
    1. scan (or enter) an existing unit code and clone it
    2. select a unit code from the history of labels scanned
    3. select product attributes and print a label (NOTE: In this option, the pieces per package can be changed before printing a label) #TODO# - is this still possible, given that this would switch the product?
    Once any of the above are done a warning message appears making sure that the user knows they are RE printing a Label and that if they want to create a production label they should use the production menu instead
  4. An "inventory Movement" option exists on the handheld. When selected, a unit can be scanned and a new location can then be scanned. This would update the inventory table with the new location #TODO# (update or new record)
  5. A Product Change menu option exists. When selected, a unit can be scanned or manually entered.
    Product attributes can be then be modified. The handheld then shows a comparison of old and new attributes.
    Once confirmed, a new unit code is created (in the Inventory Table), the old one is invalidated (and linked to the new unit code for audit purposes)
    Finally, an Adjustment record is created to reduce the previous product's inventory and increase the new one
  6. A product can be invalidated by selecting the menu option and scanning/entering the unit code
    This creates an adjustment to decrease inventory of this product Code
    If necessary, the WIP cloud of the related product is also increased
  7. Physical counts are performed at differing intervals based on mill. During this process, units are scanned and the totals are compared.
    During the process, any items unverified or not put away (location code is 0) can be fixed
    At the end, any discrepancies can be corrected using adjustments
  8. A stock transfer menu option exists. If selected, multiple units can be scanned and transferred.
    This removes the inventory from the system and creates an adjustment of type "ST"
    Since Virtual Reload inventory is not barcoded, the rest of the movement (creating inventory at the reload location) is handled in JDE when processing the adjustment

Updating JDE

Up to this point, we've downloaded master data from JDE and, contained within the barcode system, created production records, inventory records and if necessary adjustments. Of all the tables and information (WIP, inventory, audit tables, PLC data, etc) only Production and Adjustments are sent to JDE.

Requirements

  1. Ability to transfer data to JDE in a specific format at specific intervals (so that when connection is down, transactions can continue and be transferred later)

Current Solution: As with the master data from JDE, it first needs to get transferred to PTDCSQL01. A job runs every 30 min (on the hour and 30 min) to move the Adjustment and Production data to PTDCSQL01 (a slightly different Production table). From there, a job runs every 30 min (on the 10 and 40 min) on PTDCSQL01 (JDE_PostProduction) to post any unprocessed Production data to JDE.

  1. Adjustment and Production Data is transfered (formatted) on the hour and 30 min to PTDCSQL01 and from there a job runs on the 10 and 40 min to transfer it to JDE
    Below are 2 maps to explain the data mapping between the tables:
    1. Mill Adjustment > PTDCSQL01 Production Table > JDE_PostProduction
    2. Mill Production > PTDCSQL01 Production Table > JDE_PostProduction

Shipping

Scalehouse

The scalehouse system (LIMS and WeighWiz) function independently of the barcode system and provide other functionality (receipt of raw materials, etc). It is likely that integration with this software will remain.

Requirements

  1. Ability to store scale order data in a format required by LIMS
  2. Ability to receive scale ticket data from LIMS
  3. Ability to track when a shipment is closed (to update the scale ticket data)

Current Solution:

  1. ScaleOrders (a table of order/shipment data in a specific format) are extracted from JDE (LIMS pulls this data as needed)
  2. As a truck enters the scalehouse, and provides a valid order/shipment, scale ticket data is created in the local SQL Server and a scale ticket prints in the loading area
  3. After the shipment is loaded, the truck is weighed on the way out and the scale ticket data is updated

Loading a Shipment

Requirements

  1. Ability to "create" a barcode shipment by entering either Order # or Shipment #
  2. Ability to open a shipment after it's been created using either the BOL or the Order and Shipment #
  3. Ability to cancel a shipment if it has no units left on it
  4. Ability to manually enter (or scan) required truck/railcar info (weight, truck and trailer, railcar number, carrier, etc)
  5. Ability to load (either by scanning or entering manually):
    1. items on the order (standard practice)
    2. items with a hot flag (override required)
    3. substitute items (table of substitutes maintained - override required)
    4. items that aren't in inventory (requires an option to create the inventory record first)
  6. Ability to remove units from a shipment
  7. Ability to view:
    1. order notes, properties and details
    2. units loaded so far
    3. hot units loaded
    4. details/properties of a specific unit (attributes)
  8. Logic to ensure mandatory truck (truck and trailer #) or railcar info (car and seal number) are entered before closing a shipment
  9. Logic to ensure the user confirms the unit count before completing a shipment
  10. Ability to close a shipment (and automatically print a BOL locally)
  11. Ability to re-print a BOL of a closed shipment
  12. Ability to receive a shipment (and all units) back into inventory

Current Solution:

  1. From the main shipment screen on the barcode handheld, users can choose "Create a Shipment" which then prompts them to choose Rail/Truck shipment and then enter an order or shipment # on the next screen
    This creates a barcode shipment record in the local database (not to be confused with the order/shipment from the ERP)
  2. If loading is discontinued, or the user logs off, anyone can re-open the barcode shipment again to continue the loading process
    #TODO - can multiple people load units at the same time?
  3. Sometimes, barcode shipments need to be canceled (#TODO why). For this option to be selected, all units must be removed from the shipment (and put back into inventory)
  4. When creating the barcode shipment, only an Order/Shipment # are mandatory, but other information can be entered (or scanned from a scale ticket)
  5. Once a shipment is created, units can be loaded in the following way:
    1. if a scanned unit matches the item, and has no hot flag, it is loaded with no prompts
    2. if the unit matches but has a hot flag, it requires an override to load (#TODO - scenarios)
    3. if a substitute unit is going to be loaded, the user must first select an override and reason (only valid substitute products maintained in a separate table can be loaded)
    4. if a scanned unit isn't in inventory, or has a damaged label, users can easily (from the Shipment screen) select "Unit Maintenance" or "Unit Putaway" to quickly re-print a label or put a unit into inventory if something went wrong
  6. Selecting "Remove Units from Shipment" allows the user to scan a unit and have it removed and added back to inventory
  7. At any time from the shipment loading screen, users can
    1. view order notes, properties and details
    2. view a list of units loaded so far against each item
    3. view a list of only hot units loaded (if any)
    4. view details/properties of a specific unit
  8. before completing a shipment, a user is prompted for additional information for either truck (truck and trailer #) or rail shipments (car and seal #)
  9. Before completing a shipment, the user must confirm the number of units loaded (due to the nature of the business, the number of units may not match the order amount, so this is a comparison of what was physically loaded to what the system shows as loaded)
  10. Once the units are confirmed, and all additional information has been entered, the user can "Complete the shipment" which prints a BOL (locally)
  11. If necessary, users can select "Re-print a BOL" and enter the BOL # or Order/Shipment combination to reprint the original BOL
  12. If necessary, a shipment can be received back into inventory - this takes all units off the shipment and puts them back into inventory
